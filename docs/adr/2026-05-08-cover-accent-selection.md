# 封面自动强调色选择

日期：2026-05-08

## 背景

foobox 目前的高亮色跟随封面逻辑，直接取 `currentImage.GetColourSchemeJSON(1)` 返回的第一个颜色，再做很轻的明暗修正。

这个策略在黑色背景下有两个明显问题：

1. 复杂封面常会把大面积灰、米、肤色、阴影色排到第一位，导致 highlight 发灰。
2. 现有修正只管亮暗，不管饱和度，所以“低饱和但面积大”的颜色会原样进入 UI。

## 决策

本次不重做整套封面采样算法，先保留宿主提供的颜色方案接口，只在 foobox 脚本层增加一层“UI 强调色筛选”。

具体规则：

1. 先向宿主请求多个候选色，而不是只取第一个颜色。
2. 以黑底 UI 为目标，优先选择对比更强、饱和度更高、但不过亮的候选色。
3. 修正时尽量保留原始色相，只调整饱和度和明度。
4. 如果整张封面都提不出合格强调色，则退回固定备用色 `RGB(223, 48, 73)`。

## 不优先选的方案

### 自己重做像素采样 / 聚类 / 调色板算法

优点：

1. 可控性最高。
2. 不依赖宿主的候选色排序。

缺点：

1. 改动面更大，兼容风险更高。
2. 需要额外处理性能、缓存、缩略图采样精度等问题。
3. 这次的问题主要出在“候选色选择规则”而不是“完全取不到颜色”，没必要先走最重方案。

## 实施点

核心改动集中在 `script/js_panels/infoArt.js`：

1. 从 `GetColourSchemeJSON(8)` 读取候选色。
2. 对候选色做黑底强调色评分。
3. 对入选颜色做饱和度、明度修正。
4. 继续复用原有 `color_scheme_updated` 广播链路，把颜色同步给播放列表、底栏、歌词等面板。

## 后续可调项

如果后面觉得“还是太保守”或“有点太艳”，优先调这些阈值：

1. 最低饱和度 `saturationFloor`
2. 亮度上下限 `lightnessFloor` / `lightnessCeiling`
3. 对比阈值 `contrast < 2.4`
4. 低饱和惩罚 `rawHsl[1] < 0.18`
